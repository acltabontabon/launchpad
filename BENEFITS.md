# Why Launchpad Saves You Money and Headaches

A concrete, numbers-where-possible look at what Launchpad actually buys you. Three benefit levers, one unifying principle, and a realistic worked example at the end.

> Numbers in this document are illustrative estimates based on typical Anthropic / OpenAI list prices as of late 2025 and median codebase sizes. Your real savings depend on your project size, the model you use, and how often you query. The *shape* of the savings holds regardless.

---

## The unifying idea, in one sentence

> **Launchpad uses your local Ollama to do the cheap, repeatable cognitive work *before* the cloud agent sees the task.**

That single sentence underlies every benefit below. The cloud agent only pays for what only it can do (reasoning, code generation, novel synthesis). The grunt work (scanning, summarizing, structuring, interviewing) is done locally, once, for free.

## Lever 1: Project context for free

The largest hidden cost in AI-assisted coding is the **exploration tax**: every fresh conversation has the cloud agent re-discover what the project is, what stack it uses, what files exist, where the auth code lives, what conventions the team follows. Most of this work is identical session over session.

Launchpad's `scan_project` returns a structured summary the cloud agent reads in one shot.

### Token math: a typical "explore this repo" session

Assumptions:
- Medium project: ~200 source files, ~50 KB average
- "What's this project?" question requires the agent to grep + read enough to form an answer

| Action | Without Launchpad | With Launchpad |
|---|---|---|
| Agent runs `grep` / `find` for project shape | ~5 calls, ~3K tokens | 0 |
| Agent reads `README.md`, `pom.xml` / `package.json` | ~2 files, ~4K tokens | 0 |
| Agent reads representative source files | 15-25 files, ~30K-50K tokens | 0 |
| Agent reads `CLAUDE.md` (already generated by Launchpad) | n/a | ~3K tokens |
| Agent calls `scan_project` | n/a | ~2K tokens returned |
| **Total input tokens spent** | **~40K-60K** | **~5K** |

### Cost translation (Claude Sonnet 4 list pricing, $3 / M input tokens)

| Scenario | Per-session input cost |
|---|---|
| Without Launchpad | $0.12 - $0.18 |
| With Launchpad | $0.015 |
| **Savings per session** | **~$0.10 - $0.16 (8-12x cheaper)** |

That's *per session*. A developer who has 10 AI-assisted sessions a day, 200 working days a year:

| | Sessions/year | Cost without | Cost with | Annual savings |
|---|---|---|---|---|
| Single developer | 2,000 | $300 | $30 | **~$270** |
| 10-person team | 20,000 | $3,000 | $300 | **~$2,700** |
| 100-person engineering org | 200,000 | $30,000 | $3,000 | **~$27,000** |

Numbers go up with Opus / GPT-5-class models (~5x), and with longer / harder sessions where the exploration tax compounds. They go down if your team uses smaller / cheaper models. Pick the row that matches your reality.

### Where the savings come from, mechanically

```
   Cloud agent without Launchpad           Cloud agent with Launchpad
   ────────────────────────────            ───────────────────────────

   "What's this project?"                  "What's this project?"
        │                                       │
        ▼                                       ▼
   grep ./src ────► 3K tokens             scan_project("foo")
   find . -name *.java ─► 1K tokens             │
   read pom.xml ──► 2K tokens                   ▼
   read 20 files ─► 40K tokens            { stack, deps, packages,
        │                                   sample symbols }
        ▼                                       │
   forms an answer                             2K tokens
        │                                       │
        ▼                                       ▼
   ~46K input tokens spent                forms an answer
                                               │
                                               ▼
                                         ~3K input tokens spent
```

The savings are not from one big optimization. They are from cutting **the same wasted exploration** that happens at the start of every conversation.

## Lever 2: Ambiguity reduction via local interview

The second-largest hidden cost is the **clarification loop**. You give the cloud agent an underspecified task. It asks 3-5 clarifying questions, or worse, makes wrong assumptions and you spend several round-trips refactoring.

Launchpad's `/new-task` runs the interview *locally* with Ollama, then hands the cloud agent a fully-specified prompt.

### Token math: a typical "add a feature" task

Assumptions:
- Task: "add a payments endpoint"
- Without Launchpad: agent asks 4 clarifying questions; first draft has 2 wrong assumptions; refactor needed
- With Launchpad: local interview clarifies the same 4 things; first draft is correct

| Action | Without Launchpad | With Launchpad |
|---|---|---|
| Initial task description | ~200 tokens in | ~200 tokens in |
| Clarifying Q&A loop (4 rounds, ~1K each direction) | ~8K in, ~8K out | 0 (done locally) |
| First draft (model output) | ~3K out | ~3K out |
| Refactor cycle (because draft missed assumptions) | ~5K in, ~3K out | 0 |
| **Total tokens** | ~16K in, ~14K out | ~3K in, ~3K out |

### Cost translation (Claude Sonnet 4 list pricing, $3 / M input + $15 / M output)

| Scenario | Per-task cost |
|---|---|
| Without Launchpad | $0.05 (input) + $0.21 (output) = **$0.26** |
| With Launchpad | $0.01 + $0.05 = **$0.06** |
| **Savings per task** | **~$0.20 (4x cheaper)** |

At 5 non-trivial tasks per developer per day, 200 working days:

| | Tasks/year | Cost without | Cost with | Annual savings |
|---|---|---|---|---|
| Single developer | 1,000 | $260 | $60 | **~$200** |
| 10-person team | 10,000 | $2,600 | $600 | **~$2,000** |
| 100-person org | 100,000 | $26,000 | $6,000 | **~$20,000** |

### The harder-to-quantify benefit: getting it right the first time

Tokens are the easy part to measure. The bigger win is **wall-clock time and developer focus**. A task that took five round-trips with the cloud agent now takes one. That's 4 fewer context switches, 4 fewer "wait for the model to think" pauses, and a draft you can review immediately instead of guiding to correctness.

```
   Without Launchpad (cloud-driven clarification)
   ──────────────────────────────────────────────
   You: "add payments endpoint"
   ⏱  agent thinks (3s) ────────────────────────►
   Agent: "what auth?"
   You: "JWT" ──┐
   ⏱  agent thinks (3s) ────────────────────────►
   Agent: "what request shape?"
   You: "JSON with idempotency key" ──┐
   ⏱  agent thinks (3s) ────────────────────────►
   Agent: "what error handling?"
   You: "RFC 7807" ──┐
   ⏱  agent thinks (4s) ────────────────────────►
   Agent: "what about rate limiting?"
   You: "100 req/min per tenant" ──┐
   ⏱  agent thinks (12s, writes draft) ─────────►
   Agent: "[draft missing tenant context]"
   You: "you forgot tenant scoping"
   ⏱  agent thinks (8s, refactors) ─────────────►
   Agent: "[refactored]"
   ────────────────────────────────────────────
   Elapsed: ~3 minutes of context switching

   With Launchpad (local interview)
   ────────────────────────────────
   You → TUI: walk through interview (local, instant)
   You → cloud agent: paste the synthesized prompt
   ⏱  agent thinks (15s, writes correct draft) ─►
   Agent: "[draft, correct on first pass]"
   ────────────────────────────────────────────
   Elapsed: ~30s of cloud time + you control the local interview pace
```

The wall-clock saving isn't on the dashboard, but it's real, and it's why developers who use this workflow stick with it.

## Lever 3: Cross-project reference (starter -> service)

The most ambitious lever. Your team has a starter / scaffold project that encodes the right patterns. You want a new service to follow those patterns. Today you'd either copy-paste manually, or have the cloud agent re-derive everything from scratch.

With Launchpad, both projects are registered. The cloud agent can query them by name in the same conversation, and reason across them.

### Token math: "apply security patterns from the starter to the new service"

Assumptions:
- Starter project: ~50 source files
- Service project: ~30 source files
- Goal: port the security config + filter chain (2-3 files of pattern)

| Action | Without Launchpad | With Launchpad |
|---|---|---|
| Agent explores starter | reads 15-25 files (~40K tokens) | scan_project: 2K tokens |
| Agent identifies security-relevant files | grep + 5 more reads (~10K tokens) | sees them in the package summary, 0 tokens |
| Agent reads the 2-3 actually-relevant files | included above | 3 targeted reads, ~6K tokens |
| Agent explores the service | reads 10-15 files (~25K tokens) | scan_project: 2K tokens |
| Agent checks standards alignment | grep team docs, ~5K tokens | get_standards: 3K tokens (structured) |
| Agent writes the new files | ~5K out | ~5K out |
| **Total input tokens** | **~80K** | **~13K** |

### Cost translation

| Scenario | Per-task cost |
|---|---|
| Without Launchpad | ~$0.24 (input) + $0.08 (output) = **$0.32** |
| With Launchpad | ~$0.04 + $0.08 = **$0.12** |
| **Savings per cross-reference task** | **~$0.20 (~2.5x cheaper)** |

But the bigger benefit on this lever is **correctness**, not cost.

### Why correctness improves

```
   Without Launchpad
   ─────────────────
   Cloud agent reads 25 files in the starter, picks "the security ones"
   based on naming heuristics. Maybe gets it right. Maybe misses a
   subtle pattern (e.g. a custom Filter ordering convention) because
   that file wasn't in the top-25 it sampled.

   With Launchpad
   ──────────────
   scan_project returns a structured package map with sample symbols:

     com.acme.security    [files=4, samples: SecurityConfig,
                            JwtAuthFilter, JwtIssuer, TenantResolver]
     com.acme.errors      [files=2, samples: ProblemDetailHandler,
                            ApplicationException]
     ...

   Agent immediately sees ALL the security-relevant files - including
   ones with non-obvious names like TenantResolver. It then reads the
   exact set, no missed conventions.
```

The local scan catalogues everything **structurally**, so the cloud agent doesn't have to guess what's relevant from filenames alone.

### The compounding benefit across a team

If your team standardizes on one starter, every new service costs you only the local-scan time the first time. Every subsequent cross-reference (whether from the cloud agent, a CI check, or a developer browsing in the TUI) is free.

```
                       ┌─────────────┐
                       │ acme-starter│
                       │ (scanned    │
                       │  once)      │
                       └──────┬──────┘
                              │
       ┌──────────────────────┼──────────────────────┐
       │                      │                      │
       ▼                      ▼                      ▼
  ┌─────────┐           ┌─────────┐            ┌─────────┐
  │payments │           │  user-  │            │  notif- │
  │ service │           │ service │            │ service │
  └─────────┘           └─────────┘            └─────────┘

  Every new service queries the starter by name.
  Patterns stay consistent across the org without manual sync.
  Tech leads update the starter -> everyone benefits.
```

## What Launchpad does NOT save you on

Honesty matters. Here's where the math doesn't help:

| Cost | Why Launchpad can't reduce it |
|---|---|
| Code generation tokens (the actual draft) | The cloud agent still has to write the code. Launchpad only reduces the *exploration* and *clarification* overhead. |
| Long-running refactors | If a task genuinely requires writing a lot of code, the output token count is what it is. |
| Hardware: running Ollama | A 7B model uses 4-6 GB of RAM. You need the headroom on the developer's machine. |
| Model quality drift | If the local model is too small / weak for your codebase, its scans and interviews will be worse than the cloud equivalent. `qwen2.5-coder:7b` is a sensible default, but bigger projects benefit from `qwen2.5-coder:14b` or larger. |
| Bad standards | If your team's `rules.yml` is vague platitudes, the standards lever has nothing real to give the cloud agent. Garbage in, garbage rendered. |

These caveats are real. Launchpad is a leverage tool, not magic.

## How to start measuring your own savings

Two practical experiments you can run this week:

### Experiment 1: Single-session token audit

1. Pick a typical "explore a project" question.
2. Ask Claude Code without Launchpad. Note the response time and (if you can see token counts in your tool) the input token count.
3. Open the same project in the TUI, run `/init` to generate the context files. Then ask the same question with Launchpad registered.
4. Compare.

You should see a ~5-10x reduction in input tokens for the second run.

### Experiment 2: Task time-to-first-correct-draft

1. Pick a non-trivial task you'd normally ask the cloud agent.
2. Without Launchpad: time how long until the agent's draft is correct (you'll iterate).
3. With Launchpad: run `/new-task` in the TUI to interview yourself, paste the synthesized prompt, time how long until the first draft is correct.

You should see the second run be a single-pass success more often, with notably less back-and-forth.
