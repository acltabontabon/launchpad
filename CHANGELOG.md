# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]
### Added
- `/new-task` interview answer input is now multi-line. ENTER inserts a newline (was: submit); Tab submits the answer; Esc still cancels. The answer box height grew to 6 rows (~4 visible lines + borders) so multi-line answers render without clipping. Mirrors the existing TaskInputView pattern for consistency.
- `/new-task` saved task files now include an HTML-commented `<!-- launchpad-interview ... -->` appendix capturing the original task description verbatim and the full Q/A transcript. The synthesised prompt is preserved as the primary content; the appendix is invisible when rendered as markdown but readable in any editor. Provides traceability for how the synthesised prompt was derived.
- `/new-task` rule selection now does **per-tag opt-out** in addition to per-rule opt-out. When the user negates a question containing keywords from a tag domain (security/auth/login/password → `security`/`auth`/`crypto`; logging/metric → `observability`; pagination → `scale`), every rule whose scope tags overlap those domains is excluded. Previously only the specific rule whose title matched got opted out; rules like `modern-password-hashing` / `output-encoding-context-aware` / `transport-security-tls-required` would still land in Constraints for a "hello world greeting API" task even after the user said no to auth.
- `/new-task` standards pack REST rule scopes refined: `pagination-cursor-first` and `hypermedia-links-for-state` are now scoped to `rest-collection` tag (only fire for collection endpoints), and `idempotency-for-mutations` is scoped to `rest-mutation` (only fires for create/update/delete tasks). The Java classifier emits these sub-tags when the task mentions collection keywords (list/search/all/browse/page) or mutation keywords (post/put/delete/create/update/add). Single-resource read-only endpoints no longer pull in pagination / hypermedia rules.
- `/new-task` `TaskAdvisorService` is now covered by unit tests (`TaskAdvisorServiceTest`, 63 tests). Covers `extractQuestion` (Q-prefix stripping, DONE detection, line preference), `isNearDuplicateOfPrior` (Jaccard + containment + stem tolerance), `firstSentence` (abbreviation handling), `parseMarkerSections` (delimiter leniency), `classifyTaskTags` (whole-word matching + sub-tags), `scopeApplies`, `detectOptedOutRules`, `detectOptedOutTags`, `groundOutOfScope`, `groundAcceptance`, `separateGoalProseFromBullets`, `pickNextUncoveredMustRule`, `assembleFinalMarkdown` (section order, cap, overflow footer, severity sorting), and stack normalisation. Closes the only genuine test-coverage gap.
- `/new-task` removed the dead `ProjectContext ctx` parameter from `assembleFinalMarkdown`. It was vestigial from the deleted `## Context` rendering path.
- `/new-task` near-duplicate question detection now uses both Jaccard similarity AND containment ratio. Jaccard missed the common case where the new question contains the prior question verbatim plus a clarifying tail (Jaccard divides by union size, which dilutes when the new question is much longer). Containment = intersection / smaller-set-size catches "Q3 starts with the exact text of Q2 plus more words" at 100% even when Jaccard is below 0.5. Threshold 0.7.
- `/new-task` opt-out detection and near-duplicate detection now use stem-tolerant word matching. "default" matches "defaults", "validate" matches "validates" / "validated" / "validating", via prefix-overlap of 4+ characters. Previously a rule titled "Authentication and Session Defaults Are Secure" was NOT detected as opted-out when the user said "no" to a question about "authentication required by default" because `default` (singular in question) didn't exact-match `defaults` (plural in title), dropping the overlap below the 2-word threshold.
- `/new-task` finalize now grounds the `## Acceptance criteria` section against the user's actual task description + interview answers. Drops two classes of bullet: (a) quote-only fragments like `"create new api"` (the model bullet-ifying user keywords instead of producing real criteria), (b) bullets whose significant words don't appear in anything the user actually said (typically hallucinations like `"internationalization"` / `"encryption"` the model picked up from prompt-instruction vocabulary). The standard fallback bullet is whitelisted.
- `/new-task` bullet normaliser now strips multiple leading `- ` / `* ` prefixes recursively. Previously a model-emitted `- - The implemented API...` had only the outer prefix stripped, then got re-prefixed during render to become `- - The implemented API...` in the output.
- `/new-task` finalize prompt no longer enumerates specific forbidden terms ("JSON", "WebSocket", "concurrent", "internationalization", "encryption", etc.). Listing them was a textbook LLM negation failure - the model treated the don't-mention list as a vocabulary suggestion and pulled words from it into Acceptance bullets. Replaced with a positive constraint: restrict vocabulary to the user's actual words plus basic English connectors.
- `/new-task` interview and finalize prompts no longer contain CORRECT example questions / outputs. Small local models treat few-shot examples as templates to copy verbatim regardless of domain - we saw "What does the new API do - what resource does it manage?" (interview example) and "What should happen when a migration fails?" (finalize example, completely different domain) both reproduced word-for-word as questions to the user even though the user was working on a greeting API. WRONG examples were also removed because models couldn't distinguish positive from negative examples and copied phrases like "concurrent requests" / "secure connection" from the WRONG output blocks. Prompts now describe expected output abstractly with explicit anti-injection rules ("if this prompt mentions X, that is NOT part of the user's task").
- `/new-task` finalize now Java-cleans the GOAL section: leading prose is kept as the goal paragraph, trailing bullets are moved into ACCEPTANCE. Models routinely emit `GOAL\n<paragraph>\n\n- bullet\n- bullet` even when told GOAL must be prose only; the bullets get shunted to where the grounding filter can decide whether they're real acceptance criteria or hallucinations.
- `/new-task` `## Context` section removed entirely (again). Tried twice as an LLM-generated section; both times the model abstracted the example shape and invented content (e.g. "This behaviour does not apply to specific subsets, and there is no constraint on the API's performance or resource usage" - none of which the user had said). The user's decisions are already captured in the Goal and Acceptance criteria sections; a separate Context just gave the model another hallucination surface. Final structure is now Goal / Constraints / Acceptance criteria / Out of scope (when present) / Standards consulted.
- `/new-task` Constraints section now caps at 10 rules. Previously a fully-populated standards pack on a REST task pulled in 22+ rules (all security + reliability + REST rules with matching scope), overwhelming the implementing agent. New behaviour: rules are sorted by severity (must / never first, then should, then avoid) and then by the YAML `priority` field (1=critical, 2, 3...). The top 10 render as full bullets; any overflow gets a single footer line listing the remaining rule ids: `_Also applicable (N more): id1, id2, ..._`. The `Rule` record now exposes the YAML `priority` field (was previously ignored).
- `/new-task` `extractQuestion` now strips leading `Q<N>:` / `<N>.` / `**Q<N>:**` prefixes from the LLM's response. The interview history was being formatted as `Q1: ... A1: ... Q2: ...` and small models were copying that label format into their own output (e.g. emitting `Q1: Should the new user-profile endpoint require authentication?` as the literal question text).
- `/new-task` interview prompt's CORRECT examples switched from the "create new API / user-profile endpoint" domain (which the model copied verbatim for any REST task) to a refactor-domain example (migration runner failure handling). Also added an explicit "do not prefix your question with `Q<N>:`" instruction since the history formatter uses those labels.
- `/new-task` Result view scrolling now uses 3-line steps instead of 1 (single-line steps were invisible against word-wrapped content where one rendered line spans 2-3 terminal rows). Added `PageUp` / `PageDown` (15-line jump) plus `Home` / `End` to snap to top / bottom. Hint bar updated.
- `/new-task` interview now enforces minimum coverage of applicable `[must]`-severity standards from the Java side. When the model returns `__DONE__` before reaching 3 rounds AND there is at least one applicable must-rule whose title hasn't been touched in any prior question, Launchpad overrides the `__DONE__` and synthesises a deterministic standards-driven question from the next uncovered rule ("Regarding the team's `<rule title>` standard - how should this apply to your task...?"). Small local models routinely emitted `__DONE__` after a single discovery question even with auth/rate-limit/error-envelope rules in the menu; prompt-side guidance alone wasn't enough. Picks rules in severity order (must first), respects opt-outs and scope, never loops past 3 rounds.
- `/new-task` finalize CORRECT example replaced with a refactor-task example deliberately in a different domain than typical user input. The old example used "create new api" / "hello world" / "no auth" / "no rate-limit" - which exactly matched many real user inputs and caused the model to copy the example's CONTEXT and ACCEPTANCE bullets verbatim (hallucinating "no authentication required" / "no rate-limit applied" when the user had never said either). New example uses "refactor the migration runner so failures don't block startup" - a totally different domain that the model can't plausibly copy when the user is asking about an API. An explicit "the example shows the SHAPE, not the CONTENT" reminder follows the example.
- `/new-task` marker parser is now lenient on delimiter variations: matches `={3,}` or `-{3,}` on either side of the marker name (so `===OUT_OF_SCOPE===`, `---OUT_OF_SCOPE---`, and `===OUT_OF_SCOPE---` all parse). Previously a single asymmetric delimiter (which models routinely emit) caused the marker to be missed and two sections to silently bleed into one - producing output where hallucinated "concurrent requests" / "internationalization" bullets appeared inside Acceptance criteria instead of getting filtered by the Out-of-scope grounding pass.
- `/new-task` interview prompt now explicitly requires walking applicable [must]-severity standards before allowing `__DONE__`. For an HTTP API task that means asking about authentication, rate-limiting, error envelope, versioning, and idempotency at minimum. The previous "Err on the side of __DONE__" guidance was being read too conservatively by small local models, which were stopping after a single discovery question instead of probing the standards menu. The new wording is direct: outputting `__DONE__` after only one round when applicable standards remain unaddressed is explicitly named as the bug we're preventing.
- `/new-task` `firstSentence` rule-content extractor no longer splits at abbreviations. Previously the `". "` substring search matched the period in "e.g.", "i.e.", "etc.", "vs." and truncated rules like `api-versioning-explicit` mid-sentence (cutting off the `e.g. /v1/...` example entirely). Now it splits on `". "` followed by an uppercase letter - real sentence boundaries only.
- `/new-task` task-tag classifier now emits `http` and `delivery` alongside `rest` whenever the task matches REST keywords. This lets skills/checklists scoped to those tags (e.g. the `add-endpoint` skill with `tags: [delivery, http]`) actually apply to REST tasks. Previously only `[rest, feature]` was emitted and many delivery-flavoured skills got excluded despite being directly applicable.
- `/new-task` Constraints bullets are now sorted by severity (`must` -> `never` -> `should` -> `avoid` -> other) instead of YAML file order, so the reader sees mandatory rules first. Stable within each severity bucket.
- `/new-task` output restructured into a compact prompt shape: `## Goal` / `## Context` / `## Constraints` (one-line bullets) / `## Acceptance criteria` / `## Out of scope` (only if non-empty) / `## Standards consulted` (skill + checklist ids at bottom). Rules now render as single-line bullets with a `[severity]` prefix, the first sentence of the rule's content (the actionable directive), and the rule id as a `*(rule-id)*` suffix - replacing the ~5-line `### Heading + description + _Why:_` block per rule. For the typical REST task with 8-10 applicable rules, the Constraints section shrinks from ~50 lines to ~10 while preserving severity, traceability, and the actionable content. The `## Out of scope` section is omitted entirely when there's nothing grounded to put in it.
- `/new-task` task-tag classifier now tokenises into whole words before checking matches, instead of substring-matching. Previously the task "create new api" triggered the `ai` tag because "ai" is a substring of "api", which then pulled every Spring AI rule into the Constraints section even though the task had nothing to do with AI.
- `/new-task` rendered rules no longer include the `_Why:_` rationale line. Rationales explain the THINKING behind a rule and exist for the human tech lead deciding whether to ship it; the downstream implementing agent just needs to know WHAT to do. Dropping rationales cuts Constraints section size roughly in half with zero loss of actionable signal.
- `/new-task` Out-of-scope bullets now get post-processed: any bullet whose significant words don't overlap either a user-negated answer's question or an opted-out rule's title is dropped. Previously the model kept inferring things like "Adding support for concurrent requests" into Out-of-scope when the user only said "no rate-limit" - those bullets now get filtered out before assembly. If the user opted out of nothing, the section is cleared rather than populated with invented exclusions.
- `/new-task` Acceptance criteria / Out-of-scope bullet normaliser now treats lines without a `- ` / `* ` prefix as CONTINUATIONS of the previous bullet (joined with a space) instead of new bullets. Fixes the case where the model wrapped a long bullet across two lines and the second line rendered as a separate broken bullet (e.g. "...constraints in" on one line and "the parent prompt." on the next).
- `/new-task` finalize now drops the `## Context` section entirely. The downstream agents (Claude Code, Cursor) have project file access and discover stack / framework / packages / classes themselves - embedding the scanner's view of all of that duplicated information they get for free and added noise (irrelevant class-name dumps, misleading "main: LoanApplication.java" entry-point labels when the scanner mis-detected the entry point). The standards `## Constraints` and the `## Relevant standards` IDs remain - those are the parts Claude/Cursor can't grep on their own.
- `/new-task` rule / skill / checklist selection now reads the YAML `scope` block (frameworks, languages, tags, tasks) authored by tech leads, instead of guessing applicability from title-word coincidence. An item applies when each non-empty scope axis overlaps with the task: framework slug ("spring-boot"), language slug ("java"), and a Java-side task classifier that maps the user's task + interview transcript into tag slugs ("rest", "ui", "security", "data", "configuration", "testing", "observability", "ai", "refactoring", "debugging", always "feature"). Previously a testing rule ("One Behavior Per Test") was leaking into REST-API task constraints because the haystack happened to contain the word "test"; with scope-based matching that rule is now correctly excluded (its `scope.tags` is `[testing]`, doesn't overlap with `[rest, feature]`). The `Rule` / `Skill` / `Checklist` records gained an optional `scope` field; YAML scopes were already authored, just not previously consumed by the Java side.
- `/new-task` views (Interview question, Result prompt, Interview transcript) now use `Overflow.WRAP_WORD` so long content wraps inside the bordered box instead of running off the right edge.
- `/new-task` rule selection for the Constraints section is now haystack-based: the user's task description is combined with the interview transcript to pick applicable rules (1+ significant title-word match is enough). For a task like "i want to create new api", every REST rule from the standards pack now lands under Constraints as the team's defaults, not just the few that the interview happened to probe. Rules the user explicitly opted out of in the interview (negation tokens: "no", "none", "skip", "don't", "without", "no need", "bypass", etc., paired with a question whose title overlaps the rule) are filtered out so the prompt does not contradict the user's stated decisions.
- `/new-task` finalize prompt gained a worked CORRECT example for brief-but-concrete answers (e.g. "hello world api returning json with greetings"). Previously the model used the "user said nothing" CORRECT example as a template even when the user DID say something briefly - emitting "the user did not specify" instead of synthesising. The new example explicitly shows the synthesis pattern. The WRONG examples were also expanded to call out inferred-exclusions ("Providing support for concurrent requests" appearing in Out-of-scope when the user only said "no rate-limit") as a separate failure mode.
- `/new-task` interview now explicitly runs in two phases: PHASE 1 discovery (basic clarifying questions about what the task IS - purpose, resource, inputs, outputs - which do NOT need to come from the standards menu and are common-sense engineering questions) and PHASE 2 standards (pick from rules/skills/checklists once the task is concrete enough to act on). For vague tasks like "i want to create new api" the model was jumping straight to standards-driven questions (rate-limit, auth) without ever asking what the API actually does; the discovery-first directive makes "what does the new API do?" the first question. The prompt includes a worked example contrasting wrong-vs-right behaviour for the exact "create new api" case.
- `/new-task` finalize prompt now has an explicit anti-hallucination guard with a worked WRONG vs RIGHT example. Previously the LLM-synthesised sections (Task / Acceptance / Out-of-scope) invented things the user never said ("WebSocket connection", "JSON format", "local database") to fill bullets when the interview was thin. Now the model is instructed to leave sections sparse rather than fabricate decisions, and the example shows it preserving the user's actual ambiguity (e.g. "the user did not specify the API's purpose - the implementing agent should confirm before building").
- `/new-task` finalize is now hybrid: the LLM only synthesises three sections that genuinely need natural-language synthesis (`## Task` summary, `## Acceptance criteria`, `## Out of scope`) via a small marker-delimited prompt; Java deterministically assembles `## Context` (from the scanner), `## Constraints` (full text of filtered rules embedded verbatim), and `## Relevant standards` (filtered skill/checklist IDs). This eliminates three failure modes that small local models kept hitting: hallucinating constraints from training-data memory rather than the passed-in rules, rendering template placeholders like `skill:<id>` literally instead of filling them, and asserting decisions the user didn't make. Filter behaviour for "what's relevant" is unchanged (transcript word-overlap from prior round).
- `/new-task` finalize step now passes only standards items the interview actually touched. Previously every rule/skill/checklist was bundled into the finalize prompt, which on a populated pack (60+ rules) overran Ollama's default 4096-token context and silently truncated the *instructions* off the end - the model then returned meta-responses like "It seems like you didn't provide a specific text input". Items are now filtered by Jaccard-style transcript matching: if a rule's id or 2+ significant title words appear in the interview transcript, it's included. Skills/checklists pass through as compact id-only lines since the design only cites them by id under "Relevant standards" anyway. The finalize template was also restructured so the required-output instructions sit at the FRONT of the prompt (survive truncation) rather than the tail.
- `/new-task` interview now detects near-duplicate questions via Jaccard similarity on significant-word sets. Small local models routinely re-ask the same question in slightly different wording across turns (e.g. three variants of "what rate-limit should the endpoint use?" after the user answered "no limit"). When the new question shares 50%+ of significant words with any prior question in the transcript, it's treated as the model having nothing new to ask and the interview finalizes. The prompt template also gained a worked WRONG example showing the re-ask anti-pattern.
- `/new-task` interview now feeds the model a compact one-line-per-item menu of standards (`rule:<id> - <title> [severity]`, `skill:<id> - <title>`, `checklist:<id>/<item-id> - <text>`) instead of dumping every rule's full description and rationale. With a populated standards pack (60+ rules), the verbose form overwhelmed small local models and caused meta-questions like "What is the purpose of these engineering guidelines?". The interview prompt is also explicitly framed as a menu-selection task with forbidden patterns spelled out (no meta-questions about the standards, no invented services). Full standards text is still passed at the finalize step where the model needs it to embed constraints.
- `/new-task` finalize step now streams the synthesised prompt from Ollama and renders chunks live in the result view (the markdown materialises as it's generated instead of leaving the user staring at a spinner for 30+ seconds). The view auto-scrolls to follow the tail while streaming, then unlocks manual scroll when the stream completes. Elapsed-time `(Xs)` suffix on every "thinking..." status. LLM errors are now surfaced explicitly with a red header + body banner; truncation / context-window errors get a hint ("the prompt may have exceeded Ollama's loaded num_ctx — try a model with a larger context or simplify the task"), and connection refused gets a "Ollama may not be running" hint. The finalize prompt's codebase-context budget is tightened to 3000 chars so it fits comfortably in a 4096-token Ollama default.
- `/new-task` slash command on the Welcome screen, replacing the `/new-feature` placeholder. Runs scan-only on the chosen project (skips Target Select and Review), then drives a local-Ollama-powered interview whose questions are sourced from the team's engineering standards (rules, skills, checklists), not from the codebase scan - intentional, to keep the model from hallucinating connections to random packages it sees. The interview asks one focused question per turn, stops when the model emits `__DONE__` (no fixed round count, safety cap at 15), and is skipped entirely when the project has no standards configured. Only the stack one-liner (e.g. "Spring Boot / Java / Maven") is fed into the interview prompt; the full codebase scan is reserved for the finalize step, where it grounds the synthesised prompt's `## Context` section with real file paths. The synthesised prompt is structured as `## Task` / `## Context` / `## Constraints` (full text of applicable rules) / `## Relevant standards` (skill + checklist IDs) / `## Acceptance criteria` / `## Out of scope`, displayed in the TUI and written to `<projectRoot>/.launchpad/tasks/<timestamp>-<slug>.md` for re-use. Press `F1` to finalize early; `Esc` cancels; `n` on the result screen starts another task on the same scanned project without re-scanning.
- Framework-aware `StackProfile` replaces the legacy `String detectedStack`. Scanner now distinguishes language / build tool / framework (Spring Boot, Next.js, Django, FastAPI, Rails, Quarkus, NestJS, Express, React, Vue, Svelte, Gin, Echo, Fiber, etc.) by parsing the build file rather than just naming the build tool. Header shows "Spring Boot / Java / Maven" instead of "Java / Maven".
- Real `DependencyExtractor` parses full build files instead of regex over the first 60 lines. Handles pom.xml (XML DOM), package.json (Jackson, with runtime / dev / peer scopes), requirements.txt, pyproject.toml (PEP 621 + Poetry), Cargo.toml, go.mod, Gemfile. Each dep carries `name@version (scope)`.
- `StructureSummarizer` groups source files into top-level packages and extracts up to 8 public symbol names per group (classes, functions, exports). The LLM prompt now sees "users/ has 4 files exporting UserController, UserService, UserRepository, User" instead of a flat dump of every source path.
- `GitIgnoreFilter` reads `.gitignore` at the project root and skips matching files / directories during the walk, on top of the existing hardcoded `SKIP_DIRS`.
- Stack-aware prompt templates under `src/main/resources/prompts/{summary,skills,cursor-rules}/{generic,spring,next,django,fastapi,rails}.txt`. `PromptSelector` picks the template based on `StackProfile.framework`, falling back to `generic` when none matches. Each framework template names specific things to look for (controllers / @SpringBootApplication for Spring, app router vs pages for Next, etc.).
- `OutputValidator` checks every LLM response for empty / suspiciously short content, missing prompt-required headings, and hallucinated file references (backtick-quoted paths that don't exist in the scanned project). On format failure, `ContextGeneratorService` retries once with a "follow the format" reminder; remaining warnings surface as a yellow banner in the Review screen.
- Existing-context awareness: when the target project already has a `CLAUDE.md` or `.cursorrules`, the scanner reads the first 800 chars and the prompt includes an "Existing Context (already documented - don't duplicate)" section so the model knows what's already covered.
- Write safety in the Review screen. Each generated file has a per-file action chip (`NEW` / `OVERWRITE` / `MERGE` / `SKIP`) and a per-file keybinding to change it (`o` overwrite, `m` merge, `x` skip). Press `d` to toggle a unified-style line diff vs the existing file. Default is `MERGE` when the existing file carries Launchpad's `<!-- launchpad:managed:start/end -->` markers (replaces only the managed block), `SKIP` when it doesn't (don't clobber hand-edited content), `WRITE_NEW` when the file doesn't yet exist. Primary CONTEXT files (CLAUDE.md / .cursorrules / adapter-driven entry point) are wrapped in managed markers on first write so subsequent runs merge in place.
- `WriteService` backs up any file it would change to `<projectRoot>/.launchpad/backups/<timestamp>/<relpath>` before writing. Save status shows the backup path.
- Eval harness under `src/test/java/com/acltabontabon/launchpad/eval/`. Four fixture projects (`src/test/resources/fixtures/{spring-boot,nextjs,fastapi,rust-cli}`) drive a parameterised `ScannerEvalTest` that asserts framework, build tool, entry point, dependency names, and extracted public symbols per fixture. Plus unit tests for `OutputValidator`, `MergeMarkers`, `FilePlan`, and `WriteService`. All run on `./mvnw test`. A separate `GenerationEvalIT` runs the full LLM pipeline against a local Ollama, gated behind `./mvnw test -Peval`.
- Launchpad-aware detection surfaced in the header. When the selected project contains a `.launchpad/standards/` directory, the header shows an `✨ launchpad-aware` badge live as the user types the path on Project Select (recomputed via a cheap `Files.isDirectory` stat on every keystroke), and persists through Target Select / Scanning / Review. Gives immediate visual confirmation that per-project YAML overrides will be applied. Previously this signal was silent: `StandardsLoader` would apply the overrides but the user only discovered it by reading the generated output.
- Inline path autocomplete on the Project Select screen. As you type, a dimmed ghost suffix shows the next matching directory; press `Tab` (or `Right`) to accept it and continue into the chosen subdirectory.
- Runtime Ollama configuration via TUI settings screen (press `c` from the Welcome screen). Persists to `~/.launchpad/config.properties` and hot-reloads the live `ChatClient` without restarting the app.
- Engineering standards (rules and workflow skills) are now defined in YAML rather than hardcoded Java. Bundled defaults ship inside the JAR at `standards/rules.yml` and `standards/skills.yml`; teams override per project by checking `.launchpad/standards/*.yml` into the target repo.
- Curated workflow skills now generate as real Claude Code skills: one `.claude/skills/<id>/SKILL.md` file per entry in `skills.yml`, with proper frontmatter (`name`, `description`). Skills become invocable as `/<skill-id>` in Claude Code instead of being descriptive prose inside `CLAUDE.md`. The CLAUDE.md `## Skills` section now points at `.claude/skills/` and only retains an inline `### Project-Specific Skills` subsection for LLM-generated suggestions.
- Dedicated `.cursor/rules/skills.mdc` output for the Cursor target, carrying the same curated skills (Cursor has no invocable skill concept, so they remain rule-style references).
- Remote git source for engineering standards. Set `launchpad.standards.remote.url` (via `/settings` or `~/.launchpad/config.properties`) and Launchpad clones the repo into `~/.launchpad/standards-cache/<hash>/` with a 1-hour TTL. Falls back to the per-project `.launchpad/standards/` override when the cache is unavailable; auth uses the user's existing git credentials (SSH agent, credential helper, gh CLI).
- Welcome screen badge for the remote standards source state (synced / offline using cache / fetch failed / local only), mirroring the Ollama readiness badge.
- Manifest-aware standards pack format. Source directories with a `standards-pack.yml` resolve through the manifest's `includes` lists, allowing rules/skills/checklists/prompts to be split across many YAML files. Legacy flat `rules.yml` / `skills.yml` at the source root still works for per-project overrides.
- Checklists and reusable prompt templates as new standards kinds, loaded from the manifest's `checklists:` and `prompts:` includes. Generate `.ai/checklists.md` + `.ai/prompts.md` for Claude and `.cursor/rules/checklists.mdc` + `.cursor/rules/prompts.mdc` for Cursor.
- Adapter-driven primary context file. A pack's `adapters/<target>.yml` declares the entry-point file's path, the sections to embed (`rules` / `skills` / `checklists` / `prompts`), and `.mdc` frontmatter for Cursor. Secondary discoverability files (per-skill `SKILL.md`, `engineering-rules.md`, etc.) continue at hardcoded paths.
- Richer rule rendering: severity chips (`must` / `should` / `avoid` / `never`) on rule headings and italic `_Why:_` rationale lines. Rule schema accepts the new `content` field as a `@JsonAlias` for the legacy `description`.
- Richer skill rendering: `title` becomes the heading when present, and an "Expected Output" block lists each `output_expectations` entry.
- Empty-state placeholder text in generated files when no source resolves a kind, telling the user how to configure the remote URL or per-project override.
- `/settings` slash command on the Welcome screen, replacing the `c` keyboard shortcut. Includes a third field for the remote standards URL.
### Changed
- `ProjectContext` is now a structured record carrying `StackProfile`, `List<Dependency>`, `List<PackageSummary>`, and an optional `existingContextSummary`, instead of `String detectedStack` + `List<String> dependencies`. `toPromptString()` is token-budget aware (defaults to 8 000 chars), emits package summaries instead of a flat source-file dump, and adds an "Existing Context" section when the target already has a context file. Downstream consumers (`ContextTemplateEngine`, `OutputValidator`) updated.
- `ProjectScanner` uses constructor injection for its config values (`maxFileSizeKb`, `includeTestNames`) and exposes a `forTesting()` factory so the eval harness can build it without Spring. Entry-point detection now looks inside Java files for `@SpringBootApplication` and Python files for `if __name__ ==` when no name-based match exists.
- `ContextGeneratorService` returns `GeneratedOutput` (content + warnings + retried flag) instead of a raw `String`, so the Review screen can surface validation issues.
- TUI header collapsed from a 3-row wizard stepper (`Welcome|Project|Target|Scanning|Review`) to a 1-row contextual status line. It now shows `◆ Launchpad · <project> · ✨ launchpad-aware · → CLAUDE` with segments appearing as state fills in, instead of redundantly mirroring the per-view `Step X of 3` titles. Welcome and Settings keep brand-only headers. The `Step X of 3 - ...` titles on Project Select, Target Select, and Scanning screens were dropped along with the stepper, reclaiming 5 vertical rows per working screen.
- `ContextTemplateEngine` is now a Spring component that depends on a `StandardsLoader`. Rules and skills are loaded fresh per scan from the target project's override path or the bundled classpath defaults.
- Target Select screen now lists the full set of generated files for each target, including `.claude/skills/<id>/SKILL.md` for Claude and `.cursor/rules/skills.mdc` for Cursor. The hardcoded display had drifted from what the engine actually writes.
- Generated `.ai/index.md` now enumerates every curated skill as an invocable `/<skill-id>` entry alongside its trigger, giving Claude a top-level table of contents that links into `.claude/skills/`.
- `StandardsLoader` resolves standards through two source tiers (remote cache, per-project override) and two formats per tier (manifest-aware pack mode, legacy flat-file). Each kind (rules / skills / checklists / prompts / adapter) walks the tiers independently.
### Removed
- Project-info footer bar from the TUI. The selected path and target were already implied by the header stepper, and the 3 rows are now reclaimed for the active view's content area.
- `EngineeringRule` enum. The 10 principles it carried are now in `src/main/resources/standards/rules.yml` and are editable without recompiling.
- Bundled `src/main/resources/standards/rules.yml` and `skills.yml`. Launchpad now ships with no opinionated defaults; teams provide their own pack via remote git or per-project override.
