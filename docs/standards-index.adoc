= Standards-index sidecar (`.launchpad/standards.index.json`)

A deterministic, machine-readable projection of a project's resolved standards
rules, one record per rule. It is emitted alongside `AGENTS.md` during a scan,
with no model call, so it is reproducible and trustworthy. A runtime agent (or a
context-mode index) can retrieve a single rule on demand instead of carrying the
whole pack, and audit findings get a canonical registry to resolve their
`ruleId` against.

== Why

The standards pack reaches an agent as prose: `AGENTS.md` plus the rendered
`.ai/engineering-rules.md`. That is good for reading, poor for querying. The
sidecar is the structural counterpart: a stable, id-keyed list of rules an agent
can index and fetch rule-by-rule ("give me the rule behind this audit finding",
"the HIGH-severity rules for Java"). Each record is self-contained, so retrieving
one rule needs nothing else from the pack.

[source]
----
standards pack (rules)
        │
        ▼
┌────────────────────────┐
│ StandardsIndexAssembler│  Deterministic, LLM-free. Resolves rules and their
│                        │  source in one decision, sorts by priority then id,
│                        │  stamps a content hash per rule.
└───────────┬────────────┘
            │
            ▼
   .launchpad/standards.index.json   ◄── records keyed by rule id
            ▲
            │  same id is the join key
            ▼
   .launchpad/audit.sarif.json       ◄── findings reference the same ruleId
   .launchpad/audit.md
----

== Generation

Built deterministically from the resolved standards by `StandardsIndexAssembler`
and persisted by `StandardsIndexStore`. Rules and their provenance come from a
single `StandardsLoader.loadResolvedRules` call, so the emitted `source`
describes exactly the emitted `rules` - they can never drift. No LLM is involved.
Generation is best-effort: a write failure is logged and never blocks a run.

== Schema

[source,jsonc]
----
{
  "schemaVersion": 1,
  "generatedAt": "2026-06-17T12:00:00Z",   // ISO-8601; NOT part of any content hash
  "source": {                               // null when no rules resolve
    "pack": "acme-standards",               // manifest id; omitted in legacy flat mode
    "version": "1.2.0",                     // manifest version; omitted in flat mode
    "origin": "local-override"              // "remote-cache" | "local-override"
  },
  "rules": [
    {
      "id": "java.no-field-injection",      // join key; matches audit.sarif.json ruleId
      "title": "No field injection",
      "severity": "HIGH",
      "scope": { "languages": ["java"], "frameworks": ["spring-boot"],
                 "tools": [], "tasks": [], "tags": [] },
      "description": "Use constructor injection ...",   // the rule's normative text
      "rationale": "Field injection hides dependencies ...",
      "source": { "pack": "acme-standards", "version": "1.2.0",
                  "origin": "local-override" },         // mirrors top-level source by design
      "auditable": true,                    // true when the rule carries a check
      "checkKind": "forbid-pattern",        // forbid-pattern | forbid-import | llm; omitted when not auditable
      "priority": 10,
      "contentHash": "<sha-256 hex>"
    }
  ]
}
----

When no rules resolve, the sidecar is still written, with `rules: []` and
`source: null`. This lets consumers tell "no standards loaded" apart from
"sidecar never generated".

=== Fields

[cols="1,3"]
|===
| `id`          | Stable rule id authored in the pack. The join key shared with audit findings.
| `title`       | Human-readable rule name.
| `severity`    | Rule severity (e.g. `CRITICAL`, `HIGH`); omitted when unset.
| `scope`       | Applicability: `languages`, `frameworks`, `tools`, `tasks`, `tags`. Empty lists mean "no restriction".
| `description` | The rule's normative text - present so a single record stands alone.
| `rationale`   | Why the rule exists; omitted when unset.
| `source`      | Provenance of the rule (pack id/version + origin). Mirrors the index-level `source`.
| `auditable`   | True when the rule carries audit-check machinery.
| `checkKind`   | The check kind when auditable; omitted for doc-only rules.
| `priority`    | Effective sort priority (lower = more important; defaults to 99).
| `contentHash` | SHA-256 over the rule's content, for drift detection.
|===

== Cross-linking with audit findings

The rule `id` is the canonical join key across artifacts:

[source]
----
standards.index.json  rules[*].id   ==   audit.sarif.json  ruleId   ==   audit.md  rule id
----

A consumer reading an audit finding can resolve its `ruleId` against this index
to recover the rule's full text, rationale, scope, and severity without parsing
the pack.

== Stability promise

* Every rule `id` is unique within the index and stable across runs of an
  unchanged pack.
* Records are ordered deterministically by `priority` then `id`.
* Each record carries a `contentHash` (SHA-256 over a labeled payload of its id,
  title, severity, scope, description, rationale, audit metadata, and source) for
  drift detection.
* Output is stable across re-runs of an unchanged pack. `generatedAt` is the only
  non-deterministic value and is deliberately excluded from every `contentHash`,
  so two runs differing only by timestamp produce identical records.
* In schema version 1, `rules[*].source` mirrors the top-level `source`. The
  duplication is intentional so a single retrieved record remains self-contained.
* `schemaVersion` is bumped on any breaking change to this shape.
