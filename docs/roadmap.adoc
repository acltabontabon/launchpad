= Launchpad Roadmap: Project Virtualization Engine
:toc:
:toclevels: 3
:icons: font
:source-highlighter: rouge

This document records the strategic direction for Launchpad: evolving from a context generator into a Project Virtualization Engine.

[[why]]
== Why this direction

Launchpad began as a token-saving context generator. That framing no longer holds when a session already has a runtime context optimizer: runtime query tools do not save tokens against a sandbox that can read files directly. Launchpad's durable value is not session plumbing; it is synthesis of understanding.

Launchpad answers one question:

[quote]
If I dropped a senior engineer into this repository for the first time, what would they need to know to become productive in five minutes?

That is the Project Virtualization Engine. It owns project virtualization, repository understanding, engineering standards, workflow discovery, task preparation, and local AI synthesis. It deliberately does not own runtime session optimization, context compression, memory across compaction, or tool-output management. Every output must encode opinion and synthesis that a generic sandbox cannot cheaply reproduce.

[[scope]]
== Scope decisions

* Spring Boot + Maven + Java first. Build extension points for other ecosystems, but ship one stack implementation.
* Files lead, the server serves. `AGENTS.md` and `.ai/*` remain the durable, vendor-neutral artifact; the MCP server serves the structured model, workflows, and standards on demand.
* Evolve in place. Refactor the existing scanner, standards, server, and `/new-task` foundation toward the new model. No big-bang rewrite.

[[model]]
== 1. Canonical ProjectContext model

The current scan snapshot is the raw substrate. A new virtualized aggregate sits on top of it, serialized to `.launchpad/project-context.json` and projected into `AGENTS.md` and `.ai/*`:

* `identity` - name, root path, primary stack, generation time, pack version, content hash.
* `architecture` - style, layers, modules, dependency map, narrative, diagram references.
* `systems` - logical subsystems: responsibility, entry points, owning packages, external dependencies.
* `workflows` - business and operational flows (see section 2).
* `standards` - detected patterns plus applied and inferred rules, skills, and checklists (see section 3).
* `operations` - run profiles, environment variables, ports, health endpoints, build and deploy commands, observability hooks.
* `documentation` - curated doc map: title, purpose tag, anchor, freshness, gaps.
* `risks` - synthesized concerns: category, severity, evidence, affected systems, suggested mitigation.

Design rules:

* Every synthesized field carries provenance (`file:line` or deterministic source) and a confidence level of `DETERMINISTIC`, `INFERRED`, or `LLM_SUGGESTED`. Trust depends on showing sources.
* Stable IDs and content hashes on every system, workflow, rule, and risk so entries cross-reference and so a downstream indexer can chunk and re-find them.
* Determinism boundary preserved: structural fields are model-free; only narrative and inference fields use the local model, with deterministic fallback.
* Extension points for polyglot later: a `StackAnalyzer` interface with the Spring scanner as one implementation.

[[workflows]]
== 2. Workflow discovery engine

Goal: infer the business and operational workflows the repository implements. This is the most differentiated and least replicable output. The pipeline is deterministic harvest, then correlate, then label:

. Signal harvest (model-free): controller endpoints, messaging and schedule listeners, persistence touchpoints, outbound integrations, and documented flows.
. Correlation: build a call and reference graph linking each trigger to service methods and to repositories or integrations; group connected components into candidate workflows with full evidence.
. Naming and summarization (bounded model call): produce a business name and a short purpose. The model labels a deterministically derived skeleton; it cannot invent steps.
. Classification: tag each workflow as inbound API, scheduled, event-driven, batch, integration, or internal.

Output: a `Workflows` section in `AGENTS.md`, a `.ai/workflows.md` file, structured entries in the model, and a `get_workflows` server tool.

[[inference]]
== 3. Engineering standards inference engine

Today standards are declared and matched. The new capability is inference: detect the patterns the codebase already follows and suggest guardrails, turning Launchpad from "applies your standards" into "discovers your de-facto standards and proposes codifying them".

* Architectural pattern detection (model-free): layering style, controller-service-repository adherence, DTO and entity separation, error-handling shape, config externalization, test conventions.
* Prevalence scoring: a pattern followed across most sites is a de-facto standard; the outlier is a candidate risk. This inverts the audit engine, which checks compliance with declared rules.
* Guardrail suggestion (bounded model call): draft a proposed rule for high-prevalence patterns that have no matching declared rule, for the tech lead to accept into the pack.
* Drift surfacing: where a declared rule exists but prevalence is low, emit a risk.

Output: `.ai/inferred-standards.md`, clearly labelled as suggested rather than enforced, plus entries in the model. Accepted inferences are promoted into the versioned standards pack, which remains the moat.

[[new-task]]
== 4. Redesigned /new-task workflow

Keep the interview, critic, validator, and finalize spine. Ground every stage in the virtualized model rather than only the raw scan and a standards menu.

* Interview seeding: match the task description against discovered workflows and systems, and inject them so questions are repository-specific.
* Critic upgrade: check that the brief names which systems it touches and acknowledges relevant risks.
* Validator upgrade: extend grounding from "grounded in the user's words" to "grounded in the project model"; reject hallucinated systems.
* Output: the final brief gains Affected Systems, Relevant Workflows, and a Risk Watchlist, making it execution-ready and reducing the cloud agent's discovery cost.

[[disposition]]
== 5. Feature disposition

Keep and harden: the deterministic scanner and Maven parsing, the standards pack system, the `/new-task` interview, and local AI synthesis with deterministic fallback and provenance.

Expand: the virtualized model, workflow discovery, standards inference, and the synthesis server tools (`get_standards`, `get_audit_findings`, `compare_standards`, plus new `get_workflows`, `get_systems`, `get_risks`, `get_project_overview`).

Reduce: server tools a sandbox can do more cheaply (`scan_project`, the documentation tools). Keep them but add "when not to call this" guidance and return references rather than inlined bodies when a runtime optimizer is present. Replace token-savings messaging with the virtualization story.

Remove: nothing structural during the first ninety days.

[[roadmap]]
== 6. Ninety-day roadmap

Ordered by leverage and differentiation.

Phase 1, days 1 to 30, foundation and repositioning::
Define and ship the virtualized model and an assembler over the existing scan, with stable IDs, content hashes, and provenance. Refactor the template engine into a projector of the model. Reframe positioning around Project Virtualization and name the runtime optimizer as the recommended companion.

Phase 2, days 31 to 55, workflow discovery::
Promote full endpoint extraction; add event, schedule, and integration signals. Build the correlation graph and candidate workflows. Add bounded-model naming, the `Workflows` output, and the `get_workflows` tool. This is the demoable proof.

Phase 3, days 56 to 80, standards inference and /new-task grounding::
Add pattern detection and prevalence scoring, guardrail suggestion with a promote-to-pack path, and drift risks. Redesign `/new-task` to seed, critique, and validate against the model.

Phase 4, days 81 to 90, synthesis server surface and proof::
Add `get_systems`, `get_risks`, and `get_project_overview`. Add reference responses to the reducible tools. Ship an end-to-end demo proving the prep, runtime, execution boundary.

[[out-of-scope]]
== Out of scope

To protect the boundary with runtime context optimizers:

* No runtime context compression, session memory, or tool-output management.
* No generic file, grep, or sandbox tooling.
* No additional ecosystem scanner implementations yet; interface only.
