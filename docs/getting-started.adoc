== Prerequisites

=== A local AI model

Launchpad uses a locally-running model for synthesis and LLM-based audit checks.
Any of the following work:

* https://ollama.com[Ollama] (recommended for simplicity)
* https://lmstudio.ai[LM Studio]
* https://github.com/ggerganov/llama.cpp[llama.cpp]
* Any OpenAI-compatible local endpoint

==== Ollama quick setup

[source,bash]
----
ollama serve
ollama pull qwen2.5-coder:7b-instruct
----

Other providers can be configured later from the `/settings` screen inside Launchpad.

== Installation

Download the binary for your platform from the https://github.com/acltabontabon/launchpad/releases[releases page]:

[cols="1,2"]
|===
|Platform |Binary
|macOS (Apple Silicon) |`launchpad-macos-arm64`
|macOS (Intel) |`launchpad-macos-x64`
|Linux (x64) |`launchpad-linux-x64`
|Windows (x64) |`launchpad-windows-x64.exe`
|===

Make it executable on macOS/Linux:

[source,bash]
----
chmod +x launchpad-macos-arm64
./launchpad-macos-arm64
----

On first launch, the Welcome screen shows a status indicator for your AI provider.
If it shows red, open `/settings` to point Launchpad at your local model endpoint.

== Generating context for a project

=== 1. Open the command palette

Press `/` on the Welcome screen and select `/init`.

=== 2. Select a project

Start typing a path or project name. The list filters as you type.
Press `Enter` to confirm. Launchpad will tell you immediately if the project is not a supported stack.

=== 3. Watch the scan

Launchpad works through three phases:

. **Analysis** -- walks the project tree, parses build files, extracts dependencies, entry points, endpoints, and documentation.
. **Audit** -- if your standards pack has `check:` rules, Launchpad runs them and writes findings to `.launchpad/audit.md`.
. **Assembly** -- builds the output file set. Narrative sections are synthesized by the local model from the structured analysis data.

Progress and a live activity log are shown on the Scanning screen.

=== 4. Review and write

The Review screen lists every file Launchpad wants to write, with an action chip per file:

[cols="1,3"]
|===
|`NEW` |File does not exist yet -- will be created.
|`MERGE` |File exists with managed-block markers -- only the marked section is updated; everything outside it is preserved.
|`SKIP` |File exists without markers -- Launchpad will not touch hand-edited content by default.
|`OVERWRITE` |File exists; you have explicitly chosen to replace it.
|===

Press `d` to diff any file against what is on disk. Press `o` to change `SKIP` to `OVERWRITE`.
Press `s` to apply all plans. Launchpad backs up any file it changes to `.launchpad/backups/<timestamp>/` before writing.

== Generated output

[source]
----
<project>/
├── AGENTS.md                          # primary context file for AI assistants
├── .ai/
│   ├── stack.md                       # detected stack and dependency summary
│   ├── engineering-rules.md           # rules from your standards pack
│   ├── checklists.md                  # checklists from your standards pack
│   └── index.md                       # index of all .ai/ files
├── .claude/skills/<id>/SKILL.md       # Claude Code slash-command stubs (one per skill)
├── .cursor/rules/<id>.mdc             # Cursor rules (if Cursor projection is enabled)
└── .launchpad/
    ├── scan.json                      # structured scan cache (read by MCP server and audit)
    ├── audit.md                       # audit findings (if standards pack has check: rules)
    └── audit.sarif.json               # SARIF 2.1.0 audit findings
----

`AGENTS.md` is the main file. It contains a project overview, architecture notes, and the engineering standards from your standards pack. The `.ai/` files break those sections out for tools that prefer smaller, focused files.

== Connecting a standards pack

A standards pack is a git repository containing your team's engineering rules, skills, and checklists in YAML. See xref:standards-packs.adoc[Standards Packs] for how to create one.

To connect it to Launchpad:

. Press `/` on the Welcome screen and select `/settings`.
. Set *Standards repo URL* to the git URL of your standards repository.
. Save. Launchpad fetches the pack at the start of each scan and caches it locally.

If the remote is unreachable, Launchpad falls back to the cached copy. If no pack is configured, context is still generated -- standards sections are replaced with a placeholder explaining how to add one.

You can also place a standards pack directly inside the project at `.launchpad/standards/`. This takes effect when no remote is configured or as a project-level override.

== Setting up the MCP server

Launchpad can wire itself into your AI tool directly from the `/settings` screen -- no manual config editing required.

. Press `/` on the Welcome screen and select `/settings`.
. Navigate to *MCP clients* and press `Enter`.
. Launchpad detects which AI tools are installed (Claude Desktop, Claude Code, Cursor) and shows a picker.
. Select the clients to connect, confirm, and Launchpad writes the MCP snippet into each tool's config file automatically. It backs up any existing config before making changes.

Restart the AI tool after connecting. The MCP server exposes the following tools:

[cols="1,3"]
|===
|Tool |What it does

|`list_projects`
|List all projects in the local registry. Accepts an optional workspace filter.

|`scan_project`
|Scan a project and return its stack, dependencies, and package structure. Returns a cached result if one exists and is younger than 24 hours.

|`get_standards`
|Return the rules, skills, and checklists from the standards pack applied to a project, as structured data.

|`get_audit_findings`
|Return rule violations from the last audit. Each finding includes rule ID, severity, file path, line number, and message.

|`get_workflows`
|Return the discovered workflows -- name, type, trigger, steps, and the systems, external calls, and data stores each touches.

|`get_systems`
|Return the logical subsystems -- name, responsibility, and owning packages.

|`get_risks`
|Return the inferred risks -- category, severity, description, affected systems, and suggested mitigation.

|`get_project_overview`
|The five-minute brief: stack, architecture, subsystems, workflows by type, build commands, top risks, and suggested guardrails, with counts. Prefer this as the first call on an unfamiliar project.

|`compare_standards`
|Diff the standards between two projects -- shows what is common, divergent, or unique to each side.

|`list_documentation`
|List the documentation pages discovered for a project with title, format, and purpose tag. Accepts an optional purpose filter (`setup`, `architecture`, `api`, etc.).

|`find_documentation`
|Search documentation pages by keyword across one project or all registered projects.

|`get_documentation`
|Read a single documentation page from a project.
|===

If you prefer to configure manually, the snippet looks like this:

[source,json]
----
{
  "mcpServers": {
    "launchpad": {
      "command": "/path/to/launchpad",
      "args": ["mcp"]
    }
  }
}
----
